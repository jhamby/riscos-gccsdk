Last updated - 13 Aug 2006

Purpose

To automate the mundane and error prone task of repeated builds, updates, packaging and sundry items of programs and libraries built using GCCSDK and its porting tools. It was originally developed as bu Peter Naulls as part of the Unix Porting Project

What the Autobuilder does is relatively straightforward, but also complex due to the diverse nature of types of build systems. Based upon a set of variables set in a configuration file, and optional helper script commands, the Autobuilder does the following for a given program:

  • Fetch the latest version of the source for the package, possibly using Debian's apt-get command. This of course presumes you're using Debian, and perform the apt-get update command regularly. If you're on a non-Debian system, we have a fetching script that knows how to fetch sources itself, although this isn't as efficient. The system currently assumes Debian testing/unstable sources.

    For other packages, the sources can also be fetched via wget, CVS or SVN.

  • Apply any Debian patches packaged with the source if possible.
  • Apply any RISC OS specific patches. These must apply correctly.
  • Run any extra commands required to set up the build correctly.
  • Configure the package, with GNU configure or xmkmf, where appropriate.
  • Call make on the sources, or the package specific rule.
  • Call make install, if requested, and generated a library package in this case.
  • Generate HTML from man pages.
  • Zip up named files, along with RISC OS application files for programs.
  • Place final package in a location for downloading, etc.

Installation

The autobuilder does not have any specific installation requirements, since it exists alongside GCCSDK. You will need to have that properly configured first. We'll assume that it's checked out into /usr/src/gccsdk, but any path is fine.

Under the Autobuilder directory are subdirectories for categories of programs, and in those are program directories, although directories can be anywhere under the Autobuilder to aid in organisation. Each program directory contains a "setvars" script, which sets variables used by the Autobuilder.

Usage

To try out the builder, create a 'build' directory somewhere (not inside the gccsdk tree) and inside it, run something like this:

   /usr/src/gccsdk/autobuilder/build -v whois

This will attempt to fetch, build and package the whois program for RISC OS, and place it in the Autobuilder's download directory.

The program is built in $PWD/whois, which will be deleted if the build is successful. If not, the directory will remain for you to examine the build failure.

The file /usr/src/gccsdk/autobuilder/whois/setvars is the file that controls the build. All these variables are documented in the setvars-template in the top level, so won't be discussed in detail here.

Packaged programs will be placed into /usr/src/gccsdk/autobuilder/autobuilder_packages

Try:

   /usr/src/autobuilder/gccsdk/build -v popt

The single line in the setvars file for this library means that "make install" will be run after a successful build. This means a build environment can be built up under /home/riscos/env.

Packaged libraries will be placed into /usr/src/gccsdk/autobuilder/autobuilder_libraries in both zip (for use natively on RISC OS) and .tgz (for use with the GCCSDK cross compiler) variants.

Non-standard locations/toolchains

It's possible to have different versions of GCCSDK set up, perhaps to target different machines or have it installed somewhere else. In this case, you will need to set environment variables so that the autobuider knows where to look. To make this easy, create a file called 'build-setvars' in the 'build' directory you created for building in to contain for example, the following:

  GCCSDK_INSTALL_CROSSBIN="/home/riscos/xscale/cross/bin/"
  GCCSDK_INSTALL_ENV="/home/riscos/xscale/env"
  AB_TARGET=-xscale

This file will be included every time the autobuilder is run from that directory to set these values automatically. GCCSDK_INSTALL_CROSSBIN in this case points to the location of a GCCSDK that is targetted at XScale. GCCSDK_INSTALL_ENV points to the 'env' dir where the include, library and other generated files are placed.

Finally, AB_TARGET is a suffix added to the names of build logs and generated archives to differentiate it from builds with other toolchains. This is optional if you only have one toolchain, but if you have several, it is essential for depenedencies to work correctly.

Other Usage

The build script accepts several flags:

  • -v Verbose. The builder echoes all operations. Otherwise it generates only success/failure output.
  • -d Debug. The temporary directory is not deleted before or at the end of the build, even if it was successful. It will also continue even if applying patches fails (to allow for them to be applied already in a previous build).
  • -p Package only. This is for testing the packaging part of the process. Note that with this option, the Autobuilder is unable to determine the version number, so the resulting zip filename will lack this.
  • -n No dependencies. By default the Autobuilder will try to build any missing dependencies of the package before building the package itself. This option prevents any dependency checking.
  • -f Don't try and fetch the sources. This allows you to unpack the sources yourself, that you've previously fetched - normally, because you're not on Debian. To use this, you will need to have a precise setup, because the Autobuilder relies on various paths to allow it find things. Example usage: 
        mkdir libmng
    cd libmng
    tar xfvz libmng-1.0.0.tgz
    (sources now unpacked into libmng-1.0.0 directory)
    /usr/src/autobuilder/build -v -f libraries/graphics/libmng

All other arguments are treated as packages to build. If none are given, then subdirectories are scanned, and all packages are built. This is likely to take several hours, but is suitable for daily cron runs.

Other Output

If a build fails, then a log of the build - "last-failure" is placed in the package directory. Upon success, "last-success" is placed in the directory, and any "last-failure" file is deleted. These files may be examined to determine failures if you don't use the verbose option.

If there is a ReadMe file in the package directory, this is concatenated onto the "header" file and added into the archive.

Patches

Any files in a package directory ending in ".p" are applied as patches using "patch -p0 -l".

The Autobuilder will also look in several standard places for Debian patches, and apply those.

Dependencies

The Autobuilder will attempt to ensure any libraries that a package depends on will be built before the package. It does this by parsing the output of the apt-cache command if available, and the contents of the depends file in the package directory, it it exists. If a dependency has a last-success file in its package directory then the Autobuilder will assume that the dependency is satisfied, if there is no last-success file then it will attempt to build the dependency and fail if it can't.

Programs in Autobuilder

The majority of programs build, but some may occasionally fail (access to Debian mirrors is not 100%), or changes to a new version, or in a dependent library may cause failures which occasionally need to be addressed.

Please contribute any fixes back to GCCSDK so they can be incorporated.

Other Issues

Some programs are built, but not packaged or installed. This gives a warning at the end of the build. This may be because it has simply not been done yet. In some cases, "make install" isn't particular appropriate - in particular, I recommend you avoid it for tk. Instead, it needs to be installed by a custom setup like tcl is. This allows us to follow Debian conventions with library locations and so forth.

You need a fast connection for this to be useful. A feature may be added for "static" building. This would allow building from archives located on a filesystem.

It does not package sources. This needs to be added to fully meet GPL requirements.
